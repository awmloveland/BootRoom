# 2.1 Collapse Dual Goalkeeper Representation — Spec
_Date: 2026-04-21_
_Umbrella: [../2026-04-21-team-building-review-design.md](../2026-04-21-team-building-review-design.md)_
_Status: Ready to implement_

## Problem

"Is this player a goalkeeper" is tracked in two places on the same object:

1. `Player.goalkeeper: boolean` — an explicit flag.
2. `Player.mentality === 'goalkeeper'` — one of four mentality enum values.

The two can disagree at runtime. For example, in `components/NextMatchCard.tsx`'s `resolvePlayersForAutoPick`, a guest flagged as a goalkeeper becomes a `Player` with `goalkeeper: true` AND `mentality: 'balanced'`. Every consumer therefore defensively checks both: `p.goalkeeper || p.mentality === 'goalkeeper'`.

This creates:
- A latent-bug surface — update one field without the other and something silently miscategorises a keeper.
- Repeated dual-check expressions throughout the codebase.
- Item 1.3's filter (`!p.goalkeeper && p.mentality !== 'goalkeeper'`) is defensively doubled specifically because of this.

## Goal

Pick one canonical representation — `mentality === 'goalkeeper'` — and eliminate `Player.goalkeeper`. Every keeper check becomes a single comparison. The DB column (if present) stays for now; translation happens at the data-load boundary.

## Fix

### Type-level changes (`lib/types.ts`)

- **`Player`**: remove the `goalkeeper: boolean` field.
- **`NewPlayerEntry`**: remove its redundant `goalkeeper?: boolean` (the existing `mentality: Mentality` field already carries the signal).
- **`GuestEntry`**: keep `goalkeeper?: boolean`. The guest UI flow doesn't expose a mentality picker, so this boolean stays as the entry-level UI signal. At resolve time it maps into the resulting `Player`'s `mentality`.

### Data-layer translation (`lib/data.ts`)

If `fetchPlayers` (or similar) reads a `goalkeeper` column from Supabase, translate on load: `mentality: row.goalkeeper ? 'goalkeeper' : row.mentality`. Writes only set `mentality` going forward. The DB column remains as a legacy field with no migration required.

### Resolve-time translation (`components/NextMatchCard.tsx`)

In `resolvePlayersForAutoPick`:
- For a `GuestEntry`, set `mentality: guest.goalkeeper ? 'goalkeeper' : 'balanced'` on the resulting `Player`. Drop the `goalkeeper:` field from the returned object.
- For a `NewPlayerEntry`, use its existing `mentality` field directly. Drop any `goalkeeper:` assignment.

### Read-site updates

Every `p.goalkeeper` read on a `Player` object becomes `p.mentality === 'goalkeeper'`. Complete list per audit (below).

## Audit procedure

Before editing, enumerate all occurrences:

```bash
grep -rn "goalkeeper" lib/ components/ app/ __tests__/
```

Categorise each hit:

1. **Player reads** (`.goalkeeper` on a `Player`) — **replace** with `.mentality === 'goalkeeper'`.
2. **GuestEntry reads** (`.goalkeeper` on a `GuestEntry`) — **keep**. Still the UI signal.
3. **Mentality enum values** (`'goalkeeper'` as string literal) — **keep**.
4. **Type definitions** (`goalkeeper: boolean` in an interface) — **remove** from `Player` and `NewPlayerEntry`; keep on `GuestEntry`.
5. **Test fixtures** (`goalkeeper: true` on a test Player-like object) — **replace** with `mentality: 'goalkeeper'` (if not already set).
6. **DB column references** (likely `lib/data.ts`) — **translate** as described above.

After editing, re-run the grep. Remaining hits should match categories 2, 3, and 6 only.

## Test plan

No new behavioural tests are required — this is a refactor. Existing tests continue to assert:

- `autoPick` puts keepers on opposing teams.
- `ewptScore` applies the GK modifier correctly.
- Variety bonus (under 1.3) excludes keepers.

All should pass unchanged after the refactor.

**Test-fixture updates** are the only test-file changes needed: replace `goalkeeper: true` with `mentality: 'goalkeeper'` in Player-like objects, and drop `goalkeeper: false` where it appears.

**Regression check:** if a test that previously relied on keeper-identification silently passes *without* the keeper being recognised (e.g., `autoPick` stops pinning GKs to opposing teams), there's a missed call site. Treat unexpected test failures as the audit backstop.

## Out of scope

- **DB schema migration.** The `goalkeeper` column (if present in Supabase) is not dropped. A future task can run a migration once runtime code no longer writes to it.
- **Adding a `mentality` field to `GuestEntry`.** Would require UI changes to the guest flow; separate UX decision.
- **Changes to the `Mentality` enum.** All four values preserved.
- **Changes to `ewptScore`, `wprScore`, or `autoPick` logic** beyond the keeper-check simplification.

- **2.2 (named constants):** land 2.1 first so "GK" helper constants reference only `mentality === 'goalkeeper'`.
