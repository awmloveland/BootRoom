# 2.7 Synthetic Player ID at Resolution Boundary — Spec
_Date: 2026-04-21_
_Umbrella: [../2026-04-21-team-building-review-design.md](../2026-04-21-team-building-review-design.md)_
_Status: Ready to implement (depends on 2.1, 2.3, 2.4)_

## Problem

`autoPick` uses `p.name` as the identity key for several operations:

- Pair matching (lookup of `guestName` → associated player).
- Count-balance filter membership (`unknownNames.has(p.name)`).
- Swap deduplication (canonical key built from sorted names per team).

If two real roster players share a name — not uncommon in an amateur league — identity comparisons silently collide. Guest naming conventions like `"Alice +1"` rely on string discipline rather than any type-level guarantee.

Name-as-identity is a latent bug: hasn't bitten yet, but the failure mode is silent (misplaced players, merged identities, impossible-to-reproduce bugs), so there's no early warning.

## Goal

Introduce a synthetic `playerId` stamped at the resolution boundary. Every identity comparison inside `autoPick` uses `playerId`. `name` stays on the Player object for UI display only.

## Prerequisites

Land after:

- **2.1** (goalkeeper representation collapse) — so `goalkeeper` isn't a field being stamped alongside.
- **2.3** (pair-pinning helper) and **2.4** (Set-based pool exclusion) — their refactored structures convert cleanly to ID-keyed comparisons.

## Fix

### Type changes (`lib/types.ts`)

Add `playerId: string` as a **required** field on `Player`. No optional marker — discipline is the point.

```ts
export interface Player {
  playerId: string   // synthetic identity stamped at resolution time
  name: string
  // ... other existing fields
}
```

### Stamping at the resolution boundary

In `components/NextMatchCard.tsx`'s `resolvePlayersForAutoPick`, stamp each constructed Player with a prefixed ID:

```ts
// Known roster player
{ playerId: `known|${known.name}`, ...known }

// Guest
{ playerId: `guest|${name}`, name, ... }

// New player
{ playerId: `new|${name}`, name, ... }

// Fallback (unknown type)
{ playerId: `unknown|${name}`, name, ... }
```

Prefixes make cross-type collisions structurally impossible: a guest named "Alice" (`guest|Alice`) and a known player named "Alice" (`known|Alice`) are distinct IDs.

### Data-layer stamping (`lib/data.ts`)

`fetchPlayers` and similar roster-load functions return `Player[]`. Each must stamp a `playerId`. Two options:

- **Preferred:** use the DB primary key — `playerId: \`roster|${row.id}\`` (genuine uniqueness even for same-named players).
- **Fallback:** use `known|<name>` if the DB key isn't easily accessible.

Document which you chose in the implementation. If you go with the fallback, a future task can switch to real DB IDs.

### `autoPick` internal changes (`lib/autoPick.ts`)

**Pair resolution at function top:**

```ts
const nameToId = new Map<string, string>(players.map((p) => [p.name, p.playerId]))
const idPairs = (pairs ?? [])
  .map(([guestName, assocName]): [string, string | undefined] | null => {
    const guestId = nameToId.get(guestName)
    if (!guestId) return null   // guest absent; ignore pair
    const assocId = nameToId.get(assocName)
    return [guestId, assocId]   // assocId may be undefined — pair logic handles it
  })
  .filter((pair): pair is [string, string | undefined] => pair !== null)
```

If two roster players share a name, `nameToId` keeps only one — that's the existing bug and is only fixable by using real DB IDs in `lib/data.ts`. The prefix scheme still prevents guest-vs-known collisions entirely.

**Pair-pinning loop — use IDs:**

Every `p.name === ...` identity check inside the loop becomes `p.playerId === ...`. The `excluded` Set from 2.4 can stay `Set<Player>` or become `Set<string>` (IDs) for a minor efficiency gain — either is acceptable.

**`findAssocTeam` helper (from 2.3) — update signature:**

```ts
function findAssocTeam(
  assocId: string | undefined,
  pinnedA: Player | null,
  pinnedB: Player | null,
  pinnedTeamA: Player[],
  pinnedTeamB: Player[],
): 'A' | 'B' | null {
  if (assocId === undefined) return null
  if (assocId === pinnedA?.playerId) return 'A'
  if (assocId === pinnedB?.playerId) return 'B'
  if (pinnedTeamA.some((p) => p.playerId === assocId)) return 'A'
  if (pinnedTeamB.some((p) => p.playerId === assocId)) return 'B'
  return null
}
```

**Count-balance filter:**

Parameter name becomes `unknownIds: Set<string>`. Membership check uses `p.playerId`:

```ts
if (unknownIds && unknownIds.size >= 2) {
  const balanced = scored.filter((s) => {
    const countA = s.teamA.filter((p) => unknownIds.has(p.playerId)).length
    const countB = s.teamB.filter((p) => unknownIds.has(p.playerId)).length
    return Math.abs(countA - countB) <= 1
  })
  if (balanced.length > 0) filteredScored = balanced
}
```

**Swap dedup:**

```ts
const key = [
  [...candidate.teamA].map((p) => p.playerId).sort().join(','),
  [...candidate.teamB].map((p) => p.playerId).sort().join(','),
].sort().join('|')
```

### Call site update (`components/NextMatchCard.tsx`)

Build `unknownIds` from the resolved Player array after `resolvePlayersForAutoPick` returns:

```ts
const guestNames = new Set(guestEntries.map((g) => g.name))
const newPlayerNames = new Set(newPlayerEntries.map((p) => p.name))
const unknownIds = new Set(
  resolved
    .filter((p) => guestNames.has(p.name) || newPlayerNames.has(p.name))
    .map((p) => p.playerId)
)

const result = autoPick(resolved, pairs, unknownIds)
```

`pairs` stays as `Array<[string, string]>` names — `autoPick` translates to IDs internally.

### Test fixture updates

In `lib/__tests__/autoPick.test.ts`, update `makePlayer` to default a `playerId`:

```ts
function makePlayer(name: string, overrides?: Partial<Player>): Player {
  return {
    playerId: `known|${name}`,
    name,
    // ... other defaults
    ...overrides,
  }
}
```

Existing name-based assertions (e.g., `onSameTeam(s, 'Alice', 'Alice +1')`) continue to work because they match players in result arrays by `name`.

Any other test file with Player fixtures needs the same treatment.

## Rating impact

None. Same players on same teams, just identified differently internally.

## Test plan

**Regression — existing suite:** all tests pass after fixture helpers add `playerId`.

**New — same-name collision:**

```ts
it('distinguishes two players with identical names via playerId', () => {
  const alice1 = { ...makePlayer('Alice'), playerId: 'known|Alice#1' }
  const alice2 = { ...makePlayer('Alice'), playerId: 'known|Alice#2' }
  const result = autoPick([alice1, alice2, makePlayer('Bob'), makePlayer('Carol')])
  for (const s of result.suggestions) {
    const count1 = [...s.teamA, ...s.teamB].filter((p) => p.playerId === alice1.playerId).length
    const count2 = [...s.teamA, ...s.teamB].filter((p) => p.playerId === alice2.playerId).length
    expect(count1).toBe(1)
    expect(count2).toBe(1)
  }
})
```

**New — guest/known name collision:**

Squad includes a known player "Alice" (`known|Alice`) and a guest "Alice" (`guest|Alice`). Both appear once across each suggestion, treated as distinct entities.

## Out of scope

- Changing the external `autoPick` API. `pairs` stays as `Array<[string, string]>` (names); translation is internal.
- Migrating name-based DB IDs to real DB primary keys throughout the app. If `lib/data.ts` already has the DB `row.id` available, use it; otherwise stamp `known|<name>` and leave the DB-ID migration to a future task.
- Changes to UI rendering. `name` remains the display field.
- Any behavioural change to the algorithm beyond identity handling.
