# 1.6 Reframe Tolerance Pool in Win-Probability Terms — Spec
_Date: 2026-04-21_
_Umbrella: [../2026-04-21-team-building-review-design.md](../2026-04-21-team-building-review-design.md)_
_Status: Ready to implement_

## Problem

`autoPick` collects a "tolerance pool" of near-best splits to populate the "Try another" button with alternatives. The current threshold (lib/autoPick.ts, around line 168) is:

```ts
s.diff <= Math.max(bestDiff * 1.05, bestDiff + 3) + 0.001
```

Two magic numbers — `1.05` and `3` — with no semantic grounding. If asked "why 5%?" there's no principled answer.

## Goal

Express the tolerance threshold in a unit that carries meaning: **how far from 50-50 are we willing to accept?** `winProbability` (lib/utils.ts:237–240) maps score difference to predicted-win probability via a logistic curve. The tolerance pool can be defined as "all splits whose score difference corresponds to a win probability within `BAND` of 0.5."

## Important — ranking does not change

`winProbability` is a monotonically increasing function of `|scoreA − scoreB|`. Minimising `|winProb − 0.5|` picks the same splits as minimising `|diff|`. **The filter's output set is effectively identical.** What changes is the threshold parameter, not the ranking.

## Fix

### Add a band-to-diff helper at the top of `lib/autoPick.ts`

```ts
const TOLERANCE_WIN_PROB_BAND = 0.095 // accept splits within ±9.5% of 50-50 win probability

/**
 * Score-diff threshold corresponding to a given win-probability band.
 * Inverts the logistic: 1 / (1 + e^(-diff/8)) = 0.5 + band → diff = -8 × ln(1/(0.5+band) - 1).
 */
function diffForBand(band: number): number {
  return -8 * Math.log(1 / (0.5 + band) - 1)
}
```

### Replace the pool threshold in `autoPick`

```ts
// Before
const pool = filteredScored.filter((s) => s.diff <= Math.max(bestDiff * 1.05, bestDiff + 3) + 0.001)

// After
const tolerance = bestDiff + diffForBand(TOLERANCE_WIN_PROB_BAND)
const pool = filteredScored.filter((s) => s.diff <= tolerance + 0.001)
```

## Equivalence to current behaviour

`band = 0.095` gives `diffForBand ≈ 3.08`, so the new threshold is `bestDiff + 3.08`.

Compared to the old rule `max(bestDiff × 1.05, bestDiff + 3)`:

| `bestDiff` | Old rule | New rule | Difference |
|---|---|---|---|
| 0 | 3.00 | 3.08 | +0.08 |
| 1 | 4.00 | 4.08 | +0.08 |
| 5 | 8.00 | 8.08 | +0.08 |
| 10 | 13.00 | 13.08 | +0.08 |
| 20 | 23.00 | 23.08 | +0.08 |
| 60 | 63.00 | 63.08 | +0.08 |
| 100 | 105.00 | 103.08 | −1.92 |

The `× 1.05` multiplier only exceeds `+3` when `bestDiff > 60` — essentially never in real 5-a-side lineups, where `bestDiff` is typically 0–5. **For practical purposes this is a pure rename.**

## Upside

- Tolerance has a meaningful unit. "Within ±9.5% of 50-50" answers "why this threshold" in one sentence.
- Prepares the ground for **item 3.1** (calibrate the logistic against real match results). Once empirical calibration lands, the band becomes grounded in actual match data rather than intuition.
- Consolidates two magic numbers into one named constant.

## Downside / risk

- Adds a small helper function and a `Math.log` call. Tiny increase in surface area.
- Very minor behavioural change when `bestDiff > 60` (pool tighter by ~2 points). Unlikely to ever fire.
- Existing tests that hardcoded the `+3` or `× 1.05` boundary need updating. Few in number.

## Test plan

**New tests in `lib/__tests__/autoPick.test.ts`:**

- **Pool contents — small bestDiff:** Set up a squad where the best split has `diff = 0`. Splits with `diff ∈ [0, 1, 2, 3]` should be in the pool; splits with `diff = 4` should not.
- **Pool contents — moderate bestDiff:** Set up a squad where the best split has `diff = 5`. Splits with `diff ∈ [5, 6, 7, 8]` should be in the pool; `diff = 9` should not.
- **`diffForBand` helper correctness:** `diffForBand(0.095)` ≈ 3.08 (±0.01); `diffForBand(0.05)` ≈ 1.61; `diffForBand(0.2)` ≈ 6.77.

**Updates to existing tests:**

- Any test that asserted the tolerance pool's exact contents based on the old `max(bestDiff × 1.05, bestDiff + 3)` rule → update expected values. The `swap deduplication` test is property-based and unaffected.
