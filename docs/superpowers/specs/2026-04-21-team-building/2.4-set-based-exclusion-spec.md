# 2.4 Replace O(n²) searchPool Filter with a Set — Spec
_Date: 2026-04-21_
_Umbrella: [../2026-04-21-team-building-review-design.md](../2026-04-21-team-building-review-design.md)_
_Status: Ready to implement (depends on 2.3)_

## Problem

Inside `autoPick`'s pair-pinning loop, every guest placement rebuilds the search pool:

```ts
searchPool = searchPool.filter((p) => p !== guest)
```

Each filter call is O(n); doing it inside the loop makes the pair-pinning phase O(n²). For practical 5-a-side squads (n ≤ 14) this is fine performance-wise, but the pattern is unclean — mutation is diffuse and reasoning about pool state mid-loop is harder than it should be.

## Goal

Replace the per-iteration `searchPool.filter(...)` with a single `excluded: Set<Player>` that accumulates across the loop. The pool gets filtered once, at the end.

## Prerequisite

**Depends on 2.3** (pair-pinning helper refactor) being merged first. The post-2.3 structure makes this change trivial; combining both refactors in one PR is harder to review.

## Fix

In `lib/autoPick.ts`, within the pair-pinning section:

### Before (post-2.3 state)

```ts
for (const [guestName, assocName] of (pairs ?? [])) {
  const guest = searchPool.find((p) => p.name === guestName)
  if (!guest) continue

  const assoc = searchPool.find((p) => p.name === assocName)

  if (assoc) {
    searchPool = searchPool.filter((p) => p !== guest && p !== assoc)
    // toggle + push
    continue
  }

  const team = findAssocTeam(...)
  if (team === 'A') {
    searchPool = searchPool.filter((p) => p !== guest)
    pinnedTeamA.push(guest)
  } else if (team === 'B') {
    searchPool = searchPool.filter((p) => p !== guest)
    pinnedTeamB.push(guest)
  }
}
```

### After

```ts
const excluded = new Set<Player>()

for (const [guestName, assocName] of (pairs ?? [])) {
  const guest = searchPool.find((p) => p.name === guestName && !excluded.has(p))
  if (!guest) continue

  const assoc = searchPool.find((p) => p.name === assocName && !excluded.has(p))

  if (assoc) {
    excluded.add(guest)
    excluded.add(assoc)
    if (pairTeamToggle) {
      pinnedTeamA.push(guest, assoc)
    } else {
      pinnedTeamB.push(guest, assoc)
    }
    pairTeamToggle = !pairTeamToggle
    continue
  }

  const team = findAssocTeam(assocName, pinnedA, pinnedB, pinnedTeamA, pinnedTeamB)
  if (team === 'A') {
    excluded.add(guest)
    pinnedTeamA.push(guest)
  } else if (team === 'B') {
    excluded.add(guest)
    pinnedTeamB.push(guest)
  }
}

// Single filter at the end of the pair-pinning phase
searchPool = searchPool.filter((p) => !excluded.has(p))
```

### Critical behaviour to preserve

- **`searchPool.find(...)` must guard against already-excluded players.** Without the `!excluded.has(p)` guard, a player set aside earlier in the loop could be matched again by a later pair. The original semantics (each successive filter reduces the pool) required this implicitly.
- **`excluded` is local to the pair-pinning phase.** After the final filter, `searchPool` is the single source of truth for the subsequent split-generation phase.
- **The final `searchPool` must be exactly equivalent** to the pre-2.4 result.

## Test plan

No new behavioural tests — pure refactor. All existing pair tests must pass unchanged:

- `autoPick — one guest+associated pair`
- `autoPick — two guests sharing one associated player`
- `autoPick — associated player is a GK`
- `autoPick — associated player not in squad`
- `autoPick — guest has goalkeeper: true`

An optional perf sanity check (1000 calls on a 14-player squad) can be run locally but shouldn't be added to CI.
