# 1.4 Randomise Odd-Player Allocation — Spec
_Date: 2026-04-21_
_Umbrella: [../2026-04-21-team-building-review-design.md](../2026-04-21-team-building-review-design.md)_
_Status: Ready to implement_

## Problem

`autoPick` in `lib/autoPick.ts` computes Team A's size as `Math.ceil(n / 2)` minus pinned slots (around line 111). When the squad size is odd (n = 11, 13, …), Team A always ends up with the extra player. Team A thus inherits:

1. A persistent numerical advantage (rotation, fresh legs).
2. A persistent `+0.5` score bonus per extra player via `ewptScore`'s `depthBonus = min((n − 5) × 0.5, 3)`.

Small per-game but structurally always on the same side.

## Goal

When n is odd, both teams should be equally likely to receive the extra player. Over many games the advantage evens out across lineups.

## Fix

In `autoPick` (around lib/autoPick.ts:111), replace the hardcoded `Math.ceil(n / 2)` with a coin-flipped choice between `ceil` and `floor` when n is odd:

```ts
// Before
const sizeA = Math.max(
  0,
  Math.min(searchPool.length, Math.ceil(n / 2) - (pinnedA ? 1 : 0) - pinnedTeamA.length),
)

// After
// When n is odd, randomise which team receives the extra slot.
const extraSlotToA = n % 2 === 0 || Math.random() < 0.5
const halfSize = extraSlotToA ? Math.ceil(n / 2) : Math.floor(n / 2)
const sizeA = Math.max(
  0,
  Math.min(searchPool.length, halfSize - (pinnedA ? 1 : 0) - pinnedTeamA.length),
)
```

When n is even, `Math.ceil(n/2) === Math.floor(n/2)`, so the coin flip is a no-op — short-circuiting on `n % 2 === 0` skips the random call cleanly.

## Interaction with pinned players

If pinned counts already load one team unevenly (e.g., two guest pairs both on Team A), the existing `Math.max(0, ...)` clamp still caps `sizeA` at 0 when the math goes negative. The odd-slot randomisation is applied before this clamp, so in extreme cases the randomisation has no visible effect — which is correct.

## Interaction with the n > 20 sampling path

The coin flip happens once per `autoPick` call, before split generation. All 500 random samples (in the n > 20 path) share the same `halfSize`. Consecutive calls to `autoPick` on the same odd squad can yield different `halfSize` values — this is intentional (two "Try another" clicks can return a 6-5 solution and a 5-6 solution).

## Test plan

**New tests in `lib/__tests__/autoPick.test.ts`:**

- **Odd-count distribution:** Call `autoPick` 100 times on an 11-player all-rated squad. Count how often Team A has 6 players versus 5. Assert both values fall between 35 and 65 — a soft bound that tolerates random variance but fails a fixed 100/0 regression.
- **Even-count invariance:** Call `autoPick` 100 times on a 10-player squad. Every result should have `teamA.length === 5` and `teamB.length === 5`.
- **Odd-count with pinned GKs:** Call `autoPick` 100 times on an 11-player squad that includes 2 goalkeepers. Over those runs the odd player should land on each team between 35 and 65 times.

**Updates to existing tests:**

- Any test that asserts a specific `teamA.length` when n is odd must relax to accept either `Math.ceil(n / 2)` or `Math.floor(n / 2)` (net of pinned counts). Existing tests asserting `teamA.length + teamB.length === n` need no change.

## Out of scope

- The depth modifier (`(n − 5) × 0.5`) itself is unchanged. Whichever team has 6 players still gets the `+0.5` — but now it's equally likely to be either side.
- Seedable RNG (item 2.6). Once 2.6 lands, this `Math.random()` call will pick up the injected RNG automatically. For now the distribution test uses soft bounds (35–65 of 100) to avoid flakiness.
- Changes to pinned-player logic, pair pinning, or GK pinning.

## Interaction with other items

- **2.6 (seedable RNG)** will let us replace the 100-run distribution test with a deterministic seeded assertion.
- No conflict with 1.1, 1.2, 1.3, 1.5, 1.6 — they touch scoring and filtering; this touches split sizing.
