# 2.2 Hoist Magic Numbers to Named Constants — Spec
_Date: 2026-04-21_
_Umbrella: [../2026-04-21-team-building-review-design.md](../2026-04-21-team-building-review-design.md)_
_Status: Ready to implement_

## Problem

`lib/autoPick.ts` and `lib/utils.ts` bury multiple tunable parameters as bare numbers mid-expression. Examples: `20` (exhaustive-vs-sampled threshold), `500` (sample count), `3` (suggestion count), weight triples like `0.60 / 0.25 / 0.15` in `wprScore`, `0.90 / 0.10` in `ewptScore` (post-1.2), `28` (rustiness days), `/ 8` (logistic scale), and more.

These are tuning knobs disguised as literals — impossible to find without grep, impossible to name in conversation, easy to mis-tune because the same literal (`3`) means different things in different places.

## Goal

Every top-level tuning parameter gets a name, grouped and documented at the top of its file. Reading the file starts with "these are the knobs" and proceeds to "here's how they're applied."

## Scope rule — what to hoist, what to leave

**Hoist** (top-level tunables):
- Thresholds and limits that could reasonably be tuned during a rebalancing pass.
- Weights that sum to 1.0 in a weighted average.
- Named bonuses and penalties (variety bonus, no-GK penalty, etc.).
- Scale factors (logistic scale, GK WPR scale).

**Leave inline** (curve-fitting / intertwined constants):
- Values that only make sense as part of a formula curve (e.g., the `0.85 + 0.03 × (played − 1)` experience-penalty ramp).
- Indices, array bounds, and small integers like `0` and `1` with standard meaning.
- Float-safety epsilons that are implementation details (unless already exposed elsewhere).

When in doubt, prefer leaving inline with a descriptive one-line comment. The goal is named tunables, not exhaustive name-everything-ism.

## Fix

### `lib/autoPick.ts` — top of file (after imports)

```ts
// --- Split search ---
const EXHAUSTIVE_THRESHOLD = 20        // n ≤ this → try every split; above, sample
const FALLBACK_SAMPLE_COUNT = 500      // random shuffles tried when n > EXHAUSTIVE_THRESHOLD
const SUGGESTION_COUNT = 3             // distinct splits surfaced in the UI

// --- Filters ---
const COUNT_BALANCE_SLACK = 1          // max |unknownA − unknownB| tolerated

// --- Tolerance pool (see item 1.6) ---
const TOLERANCE_WIN_PROB_BAND = 0.095  // accept splits within ±9.5% of 50-50 win probability
const POOL_EPSILON = 0.001             // float-safety nudge on pool boundary
```

Substitute every matching literal.

### `lib/utils.ts` — top of file (after imports)

```ts
// --- Per-player score (wprScore) ---
const WPR_PPG_WEIGHT = 0.60            // shrunk points-per-game contribution
const WPR_FORM_WEIGHT = 0.25           // recency-weighted form contribution
const WPR_RATING_WEIGHT = 0.15         // rating prior contribution (fades with games played)
const RUSTINESS_MULTIPLIER = 0.88      // applied when calendar-rusty or intermittent
const RUSTINESS_DAYS = 28              // calendar threshold for rustiness check
const MIN_RECENT_GAMES = 2             // fewer played slots in recentForm → intermittent

// --- Team score (ewptScore, post-1.2) ---
const EWPT_AVG_WEIGHT = 0.90
const EWPT_TOP2_WEIGHT = 0.10
const GK_BASE_BONUS = 0.5              // minimum GK bonus when exactly one keeper present
const GK_WPR_SCALE = 2.0               // added per unit of (gkWpr / 100)
const NO_GK_PENALTY = -1.5
const DUAL_GK_PENALTY = -1
const VARIETY_BONUS = 2
const VARIETY_MIN_MENTALITIES = 3      // post-1.3: counts outfielders only

const DEPTH_PER_EXTRA_PLAYER = 0.5

const WIN_PROB_SCALE = 8               // logistic curve scale: diff / SCALE drives the sigmoid
```

Substitute every matching literal in `wprScore`, `ewptScore`, `winProbability`, and related helpers.

### Preserved inline (examples)

Leave these as-is, with a one-line comment documenting intent if not already present:

```ts
// Experience penalty ramp: 0.85 at 1 game → 0.94 at 4 games, full weight at 5+.
if (player.played >= 1 && player.played < 5) {
  score *= 0.85 + 0.03 * (player.played - 1)
}
```

The ramp constants `0.85 / 0.03 / 1 / 5` stay inline; the comment names the shape.

## Test plan

No new tests — pure refactor. All existing tests should pass unchanged.

**Regression backstop:** `npm test` after substitution. Any failure → a substitution is wrong (likely a typo or wrong constant applied in the wrong place).

## Files changed

| File | Change |
|---|---|
| `lib/autoPick.ts` | Add constants block at top; substitute literals throughout. |
| `lib/utils.ts` | Add constants block at top; substitute literals in `wprScore`, `ewptScore`, `winProbability`. |

Co-location (constants at the top of the file they govern) is preferred over a separate `lib/scoring-constants.ts`. Each constant lives next to the function it tunes.

## Out of scope

- Hoisting the experience-penalty ramp constants (intertwined formula).
- Hoisting `winCopy` thresholds (51, 55, 62, 70) — UX tuning, separate concern.
- Hoisting `leagueWprPercentiles` fallback values (40, 50, 60) — rare-case fallback, low value.
- **Any value changes.** Literals map 1:1 to named constants.
- Any behavioural or formula changes.

## Sequencing (important)

Land **after** items 1.1, 1.2, 1.3, and 2.1. Reasons:

- **1.2** changes the `ewptScore` weights from `0.65 / 0.10 / 0.25` to `0.90 / 0.10`. Hoisting before 1.2 would capture stale weights.
- **1.3** changes the variety bonus to count outfielders only — `VARIETY_MIN_MENTALITIES = 3` implicitly refers to the post-1.3 behaviour.
- **2.1** consolidates the goalkeeper representation — hoisting before 2.1 might capture constants around a `.goalkeeper` path that's about to be removed.

If the implementer picks up this item before those land, they should STOP and raise it.

## Interaction with other items

- **1.6 (win-prob tolerance):** introduces `TOLERANCE_WIN_PROB_BAND` already as a named constant. This spec co-locates it with the other split-search constants if 1.6 placed it separately.
- **2.6 (seedable RNG):** will use `FALLBACK_SAMPLE_COUNT` in the seeded path. No conflict.
- **3.1 (recalibrate logistic):** future work retuning `WIN_PROB_SCALE`. This constant becomes the natural home for that tuning.
