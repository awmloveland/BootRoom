# 1.5 Extend Count-Balance Filter to All Unknowns — Spec
_Date: 2026-04-21_
_Umbrella: [../2026-04-21-team-building-review-design.md](../2026-04-21-team-building-review-design.md)_
_Status: Ready to implement_

## Problem

`autoPick`'s count-balance filter (lib/autoPick.ts, around lines 148–159) discards splits where one team has 2+ more "new players" than the other, spreading unknowns across teams. It only considers `newPlayerNames` — players added through the new-player flow. Guests, who are equally unknown from a rating-confidence perspective, are not counted.

When multiple guests end up pinned to one team (via pair-toggle skew, or shared associated player), the filter does nothing to balance them. If new players are also present, the filter protects new-player balance while ignoring the guest imbalance — so the team with all the guests can still have new players piled on too.

## Goal

The filter should treat any player without real stats — guest or new player — as an unknown, and try to spread all unknowns across teams. Pair-pinning constraints still take precedence; when the filter rejects all splits due to pinning, the fallback to unfiltered scored splits is preserved.

## Fix

### `lib/autoPick.ts`

Rename the `newPlayerNames` parameter to `unknownNames` and update the JSDoc:

```ts
/**
 * @param pairs - Optional array of [guestName, associatedPlayerName] pairs.
 *   Each guest will be pinned to the same team as their associated player.
 * @param unknownNames - Optional set of player names that are unknown — guests or
 *   new players. Used as a post-generation count-balance filter: splits where the
 *   unknown-player count differs by more than 1 between teams are discarded. If no
 *   split passes, falls back to the full set.
 */
export function autoPick(
  players: Player[],
  pairs?: Array<[string, string]>,
  unknownNames?: Set<string>,
): AutoPickResult
```

The filter logic itself is unchanged — only the variable name:

```ts
// After
if (unknownNames && unknownNames.size >= 2) {
  const balanced = scored.filter((s) => {
    const countA = s.teamA.filter((p) => unknownNames.has(p.name)).length
    const countB = s.teamB.filter((p) => unknownNames.has(p.name)).length
    return Math.abs(countA - countB) <= 1
  })
  if (balanced.length > 0) filteredScored = balanced
}
```

### `components/NextMatchCard.tsx`

In `handleAutoPick`, build the unknown set from both `guestEntries` and `newPlayerEntries`:

```ts
// Before
const newPlayerNameSet = newPlayerEntries.length > 0
  ? new Set(newPlayerEntries.map((p) => p.name))
  : undefined

const result = autoPick(resolved, pairs, newPlayerNameSet)

// After
const unknownNameSet = new Set<string>()
for (const g of guestEntries) unknownNameSet.add(g.name)
for (const p of newPlayerEntries) unknownNameSet.add(p.name)

const result = autoPick(
  resolved,
  pairs,
  unknownNameSet.size >= 2 ? unknownNameSet : undefined,
)
```

The `Set` naturally deduplicates. The call-site guard (`size >= 2`) is preserved; item 2.5 will later unify this with the internal guard in `autoPick`.

## Scenarios — what actually changes

**Scenario 1 — balanced guest distribution** (1 guest on A, 1 on B via pair toggle). Filter is already balanced. **No behavioural change.**

**Scenario 2 — all guests share one associated player** (3 plus-ones for Alice). Pair-pinning puts all 3 on Alice's team. Filter rejects all splits (|3 − 0| > 1), falls back to unfiltered. **No behavioural change** — pair pinning overrules either way.

**Scenario 3 — mildly unbalanced guests + new players** (2 guests pinned to A, 1 to B, plus 2 new players in the free pool).
- Before: filter balances only new players. Accepts any 1-1 split of new players.
- After: filter counts total unknowns. With 2 guests on A already, rejects splits that put both new players on A (4-1 total unknowns). Prefers splits that put new players on B.

**This is where the fix has real effect.**

## Test plan

**New tests in `lib/__tests__/autoPick.test.ts`:**

- **Mixed unknowns, unbalanced guests:** Squad of 4 rated + 3 guest-pairs (2 pinned to A via toggle, 1 to B) + 3 new players. Assert every suggestion's `|countA − countB| ≤ 1` across total unknowns (guests + new players).
- **Mixed unknowns, single count:** Squad of 4 rated + 1 guest + 1 new player. Total unknowns = 2, filter applies. Assert `|countA − countB| ≤ 1`.
- **Guest fallback path:** Squad of 3 rated (including Alice) + 3 guests all paired to Alice. Pair pinning locks all 3 guests + Alice to Team A. Assert `autoPick` still returns suggestions (filter falls back).
- **Backward compatibility:** Squad of 4 rated + 3 new players (no guests) → assertions around new-player balance remain valid with the renamed set.

**Updates to existing tests:**

- Rename the describe block `autoPick — newPlayerNames count-balance filter` → `autoPick — unknownNames count-balance filter`.
- Rename variable references (`newPlayerNameSet` → `unknownNameSet`) in existing test bodies.
- Existing assertions around new-player balance remain valid.
