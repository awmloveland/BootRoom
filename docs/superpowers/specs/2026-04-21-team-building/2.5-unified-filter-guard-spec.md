# 2.5 Unify the Count-Balance Filter Guard — Spec
_Date: 2026-04-21_
_Umbrella: [../2026-04-21-team-building-review-design.md](../2026-04-21-team-building-review-design.md)_
_Status: Ready to implement (depends on 1.5)_

## Problem

The decision "should the count-balance filter run?" is encoded in two separate places with inconsistent thresholds:

- **Caller** (`components/NextMatchCard.tsx`, inside `handleAutoPick`): passes `undefined` when the set is size 0 or 1 (effectively `size >= 2 ? set : undefined`).
- **Callee** (`autoPick` in `lib/autoPick.ts`): the internal filter only runs when `unknownNames && unknownNames.size >= 2`.

Two places guarding the same rule. The call site's conditional is redundant — the callee already handles the no-op cases.

## Goal

Single source of truth: `autoPick` owns the decision. The caller always passes the full set (or `undefined` if there truly is no set to build). `autoPick`'s internal `size >= 2` guard quietly no-ops when the set is empty or has only one entry.

## Prerequisite

Depends on **1.5 (extend count-balance filter to all unknowns)** having landed first. After 1.5, the call-site variable is `unknownNameSet`. This spec assumes that naming.

## Fix

### `components/NextMatchCard.tsx`

Drop the `size >= 2 ? unknownNameSet : undefined` ternary at the `autoPick` call site:

```ts
// Before (post-1.5)
const unknownNameSet = new Set<string>()
for (const g of guestEntries) unknownNameSet.add(g.name)
for (const p of newPlayerEntries) unknownNameSet.add(p.name)

const result = autoPick(
  resolved,
  pairs,
  unknownNameSet.size >= 2 ? unknownNameSet : undefined,
)

// After
const unknownNameSet = new Set<string>()
for (const g of guestEntries) unknownNameSet.add(g.name)
for (const p of newPlayerEntries) unknownNameSet.add(p.name)

const result = autoPick(resolved, pairs, unknownNameSet)
```

### `lib/autoPick.ts`

No logic change. The existing internal guard (`if (unknownNames && unknownNames.size >= 2)`) already handles:

- `unknownNames === undefined` → falsy short-circuit → filter no-ops.
- `unknownNames.size === 0` → `size >= 2` false → filter no-ops.
- `unknownNames.size === 1` → `size >= 2` false → filter no-ops.
- `unknownNames.size >= 2` → filter runs.

**Optional doc tweak:** add a comment clarifying that passing an empty or singleton set is valid and will no-op:

```ts
/**
 * @param unknownNames - Optional set of player names that are unknown — guests or
 *   new players. Passing an empty set or a singleton is safe (filter no-ops).
 *   Used as a post-generation count-balance filter: splits where the unknown-player
 *   count differs by more than 1 between teams are discarded. If no split passes,
 *   falls back to the full set.
 */
```

## Test plan

No new behavioural tests — same outputs as pre-2.5. Existing coverage:

- Tests that pass a full set of 2+ names → filter engages. Unchanged.
- Tests that pass a 1-name set → filter no-ops. Already verified by `autoPick — with 1 new player, does not apply filter and still returns valid suggestions`.
- Tests that pass no set → filter no-ops. Already verified by `autoPick — passes no newPlayerNames — behaviour unchanged from baseline` (rename expected post-1.5).

**Regression check:** run the full suite. All should pass unchanged.
