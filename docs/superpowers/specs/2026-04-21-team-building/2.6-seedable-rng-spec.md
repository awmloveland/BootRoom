# 2.6 Optional Seedable RNG for autoPick — Spec
_Date: 2026-04-21_
_Umbrella: [../2026-04-21-team-building-review-design.md](../2026-04-21-team-building-review-design.md)_
_Status: Ready to implement_

## Problem

`autoPick` uses `Math.random()` in three (post-1.4: four) places:

1. Shuffling the GK pool so keeper pinning randomises when 3+ GKs exist.
2. Random-sample fallback for squads larger than `EXHAUSTIVE_THRESHOLD`.
3. Shuffling the tolerance pool before picking suggestions.
4. (Post 1.4) Coin flip for which team receives the odd player.

Every call uses the global `Math.random()`. Tests that exercise these paths cannot make precise assertions and resort to statistical patterns:

- The `rating-aware` test retries `autoPick` up to **20 times** looking for a specific split.
- The 1.4 distribution test (coming) will run 100 iterations with soft bounds (35–65%).

Flakiness guards against false alarms at the cost of slow tests and loose assertions.

## Goal

Expose an optional RNG parameter on `autoPick`. Production callers pass nothing — default uses `Math.random()`. Tests pass a seeded generator for deterministic behaviour.

## Fix

### API change (`lib/autoPick.ts`)

```ts
export function autoPick(
  players: Player[],
  pairs?: Array<[string, string]>,
  unknownNames?: Set<string>,
  random?: () => number,   // ← new
): AutoPickResult {
  const rng = random ?? Math.random
  // ... existing body, with every Math.random() call replaced by rng()
}
```

### Internal substitutions

Every `Math.random()` inside `autoPick` becomes `rng()`:

```ts
// GK shuffle
gkPlayers.sort(() => rng() - 0.5)

// Random-sample fallback (n > EXHAUSTIVE_THRESHOLD)
const shuffled = [...searchPool].sort(() => rng() - 0.5)

// Tolerance pool shuffle
const shuffledPool = [...pool].sort(() => rng() - 0.5)

// (Post 1.4) Odd-player coin flip
const extraSlotToA = n % 2 === 0 || rng() < 0.5
```

### Seeded RNG helper (`lib/__tests__/helpers/seeded-rng.ts`)

New file. Simple Linear Congruential Generator — deterministic per seed, portable, adequate for test-level randomness:

```ts
/**
 * Linear Congruential Generator. Deterministic per seed.
 * Not cryptographically strong — test use only.
 */
export function seededRng(seed: number): () => number {
  let state = seed >>> 0
  return () => {
    state = (state * 1664525 + 1013904223) >>> 0
    return state / 0x100000000
  }
}
```

### Test rewrites

Convert flaky retry-based tests to deterministic seeded ones:

- **`rating-aware` test:** replace the 20-iteration for-loop with a single call using `seededRng(SEED)` where `SEED` is empirically chosen to produce the target "one strong + one weak per team" split. Assert specific composition.
- **`swap deduplication` test:** currently runs 20 times with `Math.random()`. Replace with a handful of deterministic runs using different seeds, each asserting no pair of suggestions is a team-swap of another.
- **1.4 odd-player distribution** (after 1.4 lands): can keep the 100-run statistical form as a property-style check, or tighten to a deterministic assertion with a seeded run. Either is fine.

### Production call sites — no change required

`NextMatchCard.tsx` and `LineupLab.tsx` continue to call `autoPick(...)` with 2 or 3 arguments. The optional `random` parameter defaults to `undefined` → falls back to `Math.random()`. Production behaviour is byte-identical.

## Test plan

**Helper correctness (new tests):**

- `seededRng(42)` yields the same first-5 outputs on every construction.
- Two `seededRng(42)` instances produce identical sequences.
- `seededRng(42)` and `seededRng(43)` produce different sequences.
- All outputs lie in `[0, 1)`.

**Deterministic behaviour of `autoPick`:**

- `autoPick(players, pairs, unknowns, seededRng(42))` returns identical results across repeated calls.
- `seededRng(42)` vs `seededRng(43)` on a GK-heavy squad produces different GK pinning assignments.

**Migrated retry tests:**

- `rating-aware` test is deterministic — no retry loop, single seeded run, specific split assertion.
- `swap deduplication` test uses a handful of seeds and asserts no-swap for each.

**Unchanged production-path behaviour:**

- Every existing test that does **not** pass a seed continues to pass unchanged. This is the implicit regression check on production behaviour.

## Out of scope

- Replacing `Math.random()` calls outside `autoPick` (other lib files, UI components).
- Stronger RNG than LCG (cryptographic strength not needed for test-level determinism).
- Making the seed itself a parameter — callers construct their own RNG.
- Any behavioural change to the `autoPick` algorithm.
